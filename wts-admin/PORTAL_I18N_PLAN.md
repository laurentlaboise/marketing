# Client Portal Multilingual Plan — English + Thai

**Scope:** the client portal only (`wts-admin` — `/portal` and `/portal/boards`), plus the
customer-facing emails and errors it sends. The marketing site and the admin back-office are
explicitly out of scope for this phase.

**Status:** approved and in implementation (Phases 0–4 on this branch).
**Date:** 2026-07-03

**Ratified decisions (2026-07-03):**
1. Thai dates use the **Gregorian year with Thai month names** (`th-TH-u-ca-gregory`).
2. The whiteboard **board canvas (`board.ejs`) is deferred** to a follow-up; the boards list
   page and the JSON toast errors ARE localized in this phase. (Also deferred with it: the
   Stripe line-item description in `whiteboard/payments-service.js`.)
3. Thai copy ships **AI-drafted, flagged for native review** of the top ~60 strings before
   announcing to clients; the reviewer gets screenshots of strings in place, not a flat JSON.
4. Phase 5 gains a **production missing-key audit**: browse the portal with `?lang=th` and
   harvest the `[i18n] Missing th translation` log lines — those keys are the priority list
   for the native review round.

---

## 1. What the audit found (why the plan is shaped this way)

Three parallel code audits (string surface, server architecture, front-end/Thai readiness)
established the following facts:

### Architecture
- Express 5 + EJS 6 on Node 20 (Railway/Nixpacks) → full `Intl`/ICU support, so native
  Thai date/number formatting works without adding a library.
- PostgreSQL via `pg`; schema managed by idempotent DDL in `database/db.js` run at boot
  (`DO $$ ... ADD COLUMN IF NOT EXISTS` pattern) — adding a language column is a one-block change.
- Portal auth is its own customer session (`req.session.customerId`), separate from admin
  passport auth. Sessions are stored as JSON blobs in `user_sessions` (connect-pg-simple), so a
  `session.locale` key needs **zero schema work**.
- There is **no i18n library, no `t()` helper, no locale middleware, no cookie-parser** anywhere.
- Global `res.locals` middleware in `server.js:168-198` runs before all route mounts — the natural
  hook point for `res.locals.locale` / `res.locals.t`.

### String surface (~300–330 strings total)
| Surface | ~Strings | Notes |
|---|---|---|
| 6 portal views (`src/views/portal/*.ejs`) | ~145 | login, orders (dashboard), billing, files, chat, profile |
| Whiteboard portal (`portal-list.ejs` + shared `board.ejs` + JSON errors from `collab.js`/`assets.js`) | ~120 | ~85 of these are in `board.ejs`, which is **shared with the admin board view** |
| `portal.js` route handlers (errors, titles, rate-limits, chat JSON) | ~30 | |
| `public-api.js` portal signup/login endpoints | ~6 | these mint portal sessions from the public site |
| Customer emails (magic link, order received, file ready, review requested) | ~25 | all hardcoded English in `mailer.js`, `payments.js:348`, `business.js:1117`, `collab.js:124` |
| Shared `error.ejs` rendered to customers on portal errors | 3 + partials | uses **admin** partials — a leak to fix |

### Structural traps
1. **No shared layout.** All 8 client-facing views are standalone full HTML documents. The topbar,
   6-item nav, and footer are copy-pasted into 5+ files. Translating before extracting a layout
   multiplies every change ×8.
2. **Labels derived in code.** Order/billing status labels are generated by capitalizing raw DB
   enum values (`orders.ejs:211`, `billing.ejs:103`), and there's English pluralization
   (`"N open comment(s)"`, `portal-list.ejs:74`). These need locale maps, not string swaps.
3. **`lang="en"` hardcoded 8×**, dates hardcoded to `'en-GB'` in 5 places, money formatted with
   `toFixed(2) + ' ' + currency` and a manual LAK thousands-separator regex.
4. **No Thai-capable font.** Every portal page loads Inter (Latin only); Thai text today falls
   back to inconsistent OS fonts. `board.ejs` loads no webfont at all.
5. **Inline JS with CSP nonces** carries user-visible strings (chat "Thinking…", errors, ~40
   board strings) — translations must be injected into the page, not fetched.
6. Good news: **no PDF/invoice generation exists** (invoices are manual/email), and the AI chat
   already answers in the customer's language (`strategist.js:78` system prompt) — only the UI
   chrome needs translation.

## 2. Recommended approach

**A tiny homegrown i18n module + JSON dictionaries — no framework.**

Rationale: the codebase is deliberately dependency-light and hand-rolled (no config module, no
migration framework, no CSS framework). Two locales and ~330 flat strings don't justify i18next's
configuration surface. Thai has no plural forms and no RTL, so the hard ICU features aren't
needed. A ~80-line `src/lib/i18n.js` gives us everything: dictionary loading, `t(key, vars)`
interpolation, English fallback for missing keys, and locale-resolution middleware. If the portal
later needs 5+ languages or plural-heavy locales, swapping the internals for i18next behind the
same `t()` API is a contained change.

**Locale resolution order** (first match wins):
1. `?lang=en|th` query param → also persists to session and (if logged in) to the DB
2. `req.session.locale`
3. `customers.preferred_language` (copied into the session at login so no per-request query)
4. `Accept-Language` header (`th` → Thai)
5. Default `'en'` (env-overridable via `PORTAL_DEFAULT_LOCALE`)

**Persistence:** additive `preferred_language VARCHAR(8)` column on `customers`, using the
existing DO-block pattern in `db.js` (~lines 869–885). Session for anonymous users (login page).

## 3. Implementation phases

### Phase 0 — Structural prep (no behavior change, no translations yet)
The refactor that makes everything else cheap:
1. **Extract portal partials** into `src/views/portal/partials/`:
   - `head.ejs` — doctype, `<html lang="<%= locale %>">`, meta, Google Fonts link, shared base styles
   - `topbar.ejs`, `nav.ejs`, `footer.ejs` — the copy-pasted chrome in 5 views
2. **Portal-owned error view** `src/views/portal/error.ejs` — stop rendering the admin
   `error.ejs` (which pulls admin partials and `main.js`) to customers. Update all
   `res.render('error', …)` calls in `portal.js` and `routes-portal.js`.
3. **Formatting helpers** in `src/lib/format.js`, exposed to views:
   - `formatDate(date, locale)` — replaces the 5 hardcoded `'en-GB'` call sites
   - `formatMoney(amount, currency, locale)` — `Intl.NumberFormat`; LAK stays 0-decimals
     (replaces `toFixed(2)` and the LAK regex)
   - `formatFileSize(bytes, locale)` — replaces the duplicated inline `fmtSize()`
4. **Status-label maps** — replace `status.charAt(0).toUpperCase()+…` with an explicit
   `orderStatusLabel(status, t)` lookup; same for payment-method labels and the board
   approval/`open comment(s)` labels.

*Verification: portal renders pixel-identical in English; grep confirms no remaining
`toLocaleDateString('en-GB'` or `charAt(0).toUpperCase` in portal views.*

### Phase 1 — i18n core
1. `src/lib/i18n.js`: loads `src/locales/en.json` + `src/locales/th.json` at boot; exposes
   `t(locale, key, vars)` with `{var}` interpolation and English fallback (log missing keys).
2. **Middleware** mounted on `/portal` (and applied inside the whiteboard portal router):
   resolves locale per the order above, sets `req.locale`, `res.locals.locale`,
   `res.locals.t = (key, vars) => …`.
3. **DB column** `customers.preferred_language` (nullable, values `en`/`th`).
4. `establishCustomerSession()` copies `preferred_language` → `session.locale` at login;
   `?lang=` handler writes both session and column.
5. Locale files namespaced by page: `login.*`, `nav.*`, `dashboard.*`, `billing.*`, `files.*`,
   `chat.*`, `profile.*`, `boards.*`, `errors.*`, `emails.*`, `common.*`.

### Phase 2 — String extraction (English first, portal stays English-only and identical)
1. Replace all hardcoded strings in the 6 portal views + partials with `<%= t('…') %>` and fill
   `en.json` (~145 keys).
2. `portal.js` handler strings (~30): titles, login/validation errors, rate-limit messages, chat
   JSON errors → `req.t(…)`. Same for the 6 portal-auth strings in `public-api.js`.
3. **Inline JS strings**: inject per-page translations via a nonce'd
   `<script type="application/json" id="i18n">` block (or `window.I18N = <%- JSON.stringify(…) %>`)
   so chat/board scripts read labels from it. Keeps CSP intact, no client fetch.
4. **Whiteboard**: `portal-list.ejs` fully; `board.ejs` gets `locale` passed from each route —
   the admin route passes `'en'`, so **the admin board is unaffected**. `collab.js`/`assets.js`
   JSON error responses take the requester's locale (they surface directly in portal toasts).
5. **Emails**: add a `locale` parameter to `sendMagicLink` and the 3 other customer emails;
   subject/body pulled from `emails.*` keys; locale from `customers.preferred_language`
   (default `en` for brand-new signups).
6. **Stripe Checkout**: pass `locale: 'th'` on session creation in `payments.js` when the
   customer's locale is Thai (Stripe supports it natively — receipts and checkout UI localize
   for free).

*Verification: grep gate — no user-visible string literals left in portal views/handlers;
English rendering unchanged (screenshot diff).*

### Phase 3 — Thai enablement
1. **Fonts**: add `Noto Sans Thai` (weights 400/600/700) to the single Google Fonts request in
   the shared head partial; stack becomes `'Inter','Noto Sans Thai',Arial,sans-serif`. Give
   `board.ejs` the same webfont link (it currently has none).
2. **CSS hardening for Thai** (in the shared styles):
   - Add explicit `line-height` on badges/pills (`.badge`, `.pw-state`, `.role-badge`) — Thai
     tone marks clip in tight pills.
   - Never apply `letter-spacing` or `text-transform` to translatable text (Thai has no case;
     letter-spacing breaks cluster rendering). Drop `text-transform: capitalize` from
     `.role-badge` — the label now comes translated from the locale map.
   - Review `white-space: nowrap` buttons ("Download", "Set password") with Thai strings.
3. **Dates**: use `th-TH-u-ca-gregory` (Thai month names, **Gregorian year**) — see open
   decision #1 below.
4. **Translate `th.json`** (~330 keys). Draft with AI in a consistent polite/formal register
   (customer-service ครับ/ค่ะ-neutral), then a native-speaker review pass over the ~60
   highest-visibility strings (nav, login, dashboard, billing, emails) before launch.
5. Translate the 4 email templates.

### Phase 4 — Language switcher UX
1. **Topbar/footer switcher** on every portal page (including login, pre-auth): `EN | ไทย`
   links via `?lang=` — session-persisted, DB-persisted once logged in.
2. **Profile "Preferences" card** in `profile.ejs`: a `<select>` (English / ไทย) in a third card
   following the existing card+label pattern; saved by the existing `POST /portal/profile`
   handler alongside name/company/phone.
3. `<html lang>` reflects the active locale (already wired via the head partial).

### Phase 5 — QA and rollout
1. **Playwright screenshot pass**: every portal page × {en, th} × {mobile, desktop}; eyeball for
   clipped tone marks, overflowing buttons, wrong fonts.
2. **Functional pass in Thai**: login (magic link email in Thai), place-order statuses, billing
   totals, chat errors, board comment flow, profile save.
3. **Missing-key safety**: any key absent from `th.json` renders English and logs a warning —
   the portal can never show a raw key to a customer.
4. **Rollout**: no feature flag needed — default stays English; Thai is opt-in via the switcher.
   Test in production with `?lang=th` before announcing to clients.

## 4. Open decisions (need Laurent's call, none block Phase 0–2)

1. **Buddhist vs Gregorian year for Thai dates.** Native Thai `Intl` formatting defaults to the
   Buddhist era (2569, not 2026). **Recommendation: Gregorian with Thai month names**
   (`th-TH-u-ca-gregory`) — clients cross-reference invoices/emails with international payment
   systems that all use Gregorian.
2. **Whiteboard `board.ejs` in scope now or later?** It's ~85 strings and shared with admin.
   Recommendation: do `portal-list.ejs` + the JSON toast errors in this phase (cheap, high
   visibility) and defer the full board canvas UI to a follow-up — it's the deepest, riskiest
   surface for the least-used page.
3. **Native Thai review.** AI-drafted Thai will be grammatical but a native pass on the top ~60
   strings is strongly recommended before announcing to clients.
4. **Third language later?** The architecture (locale files + `t()`) makes Lao (`lo.json`) or
   French additive — no code changes, only a dictionary + font check (Noto Sans Lao). Nothing to
   do now; just confirming nothing in this plan paints us into a corner.
